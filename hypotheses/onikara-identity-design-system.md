# Hypotheses — Onikara identity and design system

## Meta

- **Feature:** [knowledge/product/features/onikara-identity-design-system.md](../knowledge/product/features/onikara-identity-design-system.md)
- **Status:** active
- **Owner:** Founder/creator
- **Last updated:** 2026-05-24

## Value

### H-V1: A dark fantasy/Oni/tactical identity will make Onikara more memorable than generic RPG tooling

- **Origin:** proactive
- **Confidence:** medium
- **Evidence for:**
  - The selected name and visual territory emphasize Oni masks, embers, dark metal, seals/runes, elemental magic, and tactical consequence. [source/planning/BRANDING_IDEAS.md](../source/planning/BRANDING_IDEAS.md)
  - The founder explicitly wants to avoid generic SaaS and generic anime execution. `(chat, no artifact)`
- **Evidence against:**
  - No external users have validated the brand direction yet. `(chat, no artifact)`
- **Open questions / caveats:**
  - The identity must work across both dense admin interfaces and expressive player screens.
- **Test:** Run sketch review with target testers once first viewport variants are ready.
- **Decision trigger:** Promote if testers describe the experience as tactical, premium, suspenseful, and worth exploring.
- **Resolution:** Pending.

## Usability

### H-U1: Separate admin/player visual modes can preserve clarity without splitting the brand

- **Origin:** proactive
- **Confidence:** medium
- **Evidence for:**
  - Roadmap calls for sober, dense admin UI and more expressive player UI under one Onikara identity. [source/planning/ROADMAP.md](../source/planning/ROADMAP.md)
- **Evidence against:**
  - No implemented design system has proven the shared token model yet. `(chat, no artifact)`
- **Open questions / caveats:**
  - Components need accessibility, focus, loading, empty, error, disabled, and success states from the start.
- **Test:** Validate the same token/component set in one admin screen and one player screen.
- **Decision trigger:** Promote if both screens feel related but optimized for their tasks.
- **Resolution:** Pending.

## Feasibility

### H-F1: The project can define enough design system structure before full gameplay exists

- **Origin:** proactive
- **Confidence:** low
- **Evidence for:**
  - Phase 2 plans include tokens and base components before Phase 3/4 UI implementation. [source/planning/ROADMAP.md](../source/planning/ROADMAP.md)
- **Evidence against:**
  - Gameplay and admin workflows may reveal missing states/tokens later. `(chat, no artifact)`
- **Open questions / caveats:**
  - Avoid overdesigning components before real flows exist.
- **Test:** Build only components needed by the first admin/player slices.
- **Decision trigger:** Promote if the initial tokens/components survive Phase 3/4 without major rework.
- **Resolution:** Pending.

## Viability

### H-Vi1: Strong identity is worth prioritizing early because first impression is a core validation target

- **Origin:** proactive
- **Confidence:** medium
- **Evidence for:**
  - The founder wants to measure suspense, strategy, premium feel, desire to explore, and clarity for new users. `(chat, no artifact)`
- **Evidence against:**
  - Identity work does not by itself validate retention or gameplay depth. `(chat, no artifact)`
- **Open questions / caveats:**
  - Visual validation must be paired with navigable task validation.
- **Test:** Pair visual review with task-based first viewport/navigation tests.
- **Decision trigger:** Promote if visual identity improves comprehension and motivation without hurting task clarity.
- **Resolution:** Pending.

## Other risks

### H-O1: The product could drift into either generic SaaS or over-stylized anime

- **Origin:** proactive
- **Confidence:** high
- **Evidence for:**
  - Founder named this as a current high-friction product-definition risk. `(chat, no artifact)`
- **Evidence against:**
  - Current roadmap explicitly encodes separate admin and player UI direction. [source/planning/ROADMAP.md](../source/planning/ROADMAP.md)
- **Open questions / caveats:**
  - Maintain a checklist in design reviews for "not generic SaaS" and "not anime at the cost of strategy".
- **Test:** Add these as review criteria during Phase 2.
- **Decision trigger:** Promote if repeated reviews catch and correct drift.
- **Resolution:** Pending.
